# HIM-017 — Bastion: terminal no usa el ancho disponible (PTY remoto atascado en 80 cols)

```yaml

estimacion: 3 puntos
```

---

## Contexto del bug

Usuario reporta (en tablet Android): "cuando me conecto por SSH debería ocupar el ancho
disponible, no está pasando, no es responsivo."

## Causa raíz (análisis de código, alta confianza)

1. `SshSession(cols: Int = 80, rows: Int = 24)` — toda sesión nueva usa el default fijo.
2. `AppLayout.kt:135` crea `SshSession()` sin pasar el tamaño real del terminal.
3. `startConnection()` → `session.openShell()` abre el canal PTY remoto con
   `ch.setPtyColumns(cols); ch.setPtyLines(rows)` = **80×24**, antes de que xterm.js
   siquiera termine de cargar/calcular su tamaño real.
4. Cuando xterm.js calcula su ancho real vía `FitAddon.fit()` (ya arreglado en v1.1.17
   para que se recalcule bien localmente con `ResizeObserver`), el JS notifica a Kotlin:
   `onResize(cols, rows)` → `session.resize(cols, rows)`.
5. **El bug real:** `SshSession.resize()` solo llama
   `channel?.setPtyColumns(newCols); channel?.setPtyLines(newRows)` — estos son
   **setters de configuración pre-apertura** del canal (MINA SSHD), NO un mensaje
   activo de cambio de ventana. Con el canal ya abierto, esta llamada no le avisa nada
   al servidor SSH remoto.

**Efecto:** xterm.js se renderiza localmente con el ancho de la tablet (por eso no se
ve "roto" a simple vista), pero el shell remoto sigue creyendo que el terminal mide
80 columnas — por eso el wrapping de `ls`, el prompt, `git log`, etc. no aprovecha el
ancho real de la pantalla. No es un problema de CSS/Compose, es que el servidor nunca
se entera del tamaño real.

## Fix propuesto

1. `SshSession.resize()`: reemplazar `setPtyColumns`/`setPtyLines` por el método de
   MINA SSHD para notificar activamente el cambio de ventana al canal ya abierto
   (`ChannelShell.sendWindowChange(cols, rows, 0, 0)` o equivalente en la API 2.18.0 —
   se confirma el nombre exacto al compilar, ya que no se pudo introspeccionar el jar
   sin conexión en este momento).
2. Abrir el canal con el tamaño real de xterm.js en vez de 80×24 fijo: pasar
   `cols`/`rows` reales al construir `SshSession`, obtenidos del primer `onResize` de
   xterm ANTES de conectar (o, más simple: conectar con el default y forzar un
   `resize()` real inmediatamente después de `openShell()` una vez que el bridge ya
   tiene el tamaño de xterm calculado).
3. Confirmar que el `resizeCheck` interval de `index.html` (line ~118) sigue
   notificando cambios correctamente — no se toca, ya funciona del lado JS.

## Historia de usuario

```gherkin
Historia de usuario:
Como usuario de Bastion en tablet (pantalla ancha)
Quiero que el terminal remoto use el ancho real disponible
Para que ls/git log/prompts no se corten como si tuviera una pantalla angosta de 80 columnas

Escenario: Conectar en tablet usa el ancho real
  Dado que conecto a un servidor desde la tablet con pantalla ancha
  Cuando el shell remoto imprime algo que depende del ancho (ls, prompt largo)
  Entonces el formato usa las columnas reales disponibles, no 80 fijas

Escenario: Rotar pantalla o cambiar de pestaña actualiza el ancho remoto
  Dado que roto la tablet o cambio el tamaño de la ventana
  Cuando el terminal local se redimensiona
  Entonces el servidor remoto recibe la notificación de cambio de ventana
  Y `stty size` en el shell remoto refleja el tamaño real
```
