# HIM-011 — Bastion: mantener sesiones SSH vivas en segundo plano

```yaml
id: HIM-011
feature: keep-sessions-alive
prioridad: alta
estimacion: 8 puntos
estado: en-progreso
dependencias: [HIM-008, HIM-010]
```

---

## Contexto

El usuario reporta que "la sesión se cierra todo el tiempo; debe mantenerse abierta incluso si abro
otras apps". El stack trace capturado (v1.1.15) confirma el síntoma:

```
java.io.IOException: Pipe closed
  at java.io.PipedOutputStream.write
  at com.bastion.app.ssh.SshSession.write
  at TerminalTab (input loop)
```

O sea: se escribe en el stdin de una sesión **ya muerta** → pipe cerrado.

### Causas raíz (verificadas en código)

1. **Sin keepalive SSH** (`SshClientManager` no configura heartbeat): una conexión inactiva la corta
   el servidor / NAT / router tras un timeout — típico al dejar la app en segundo plano.
2. **Sin foreground service**: al pasar a segundo plano, Android puede **matar el proceso** (la app
   es pesada: varios WebViews). Al morir el proceso, mueren todas las sesiones SSH.
3. No hay handler destructivo en `MainActivity` (onPause/onStop no cierran nada) — descartado.

---

## Solución

| # | Cambio | Efecto |
|---|--------|--------|
| S1 | Heartbeat SSH (`CoreModuleProperties.HEARTBEAT_INTERVAL` ~30s) + TCP keepalive | La conexión no muere por inactividad al cambiar de app |
| S2 | Foreground service con notificación mientras haya ≥1 terminal activa | Android no mata el proceso en segundo plano → las sesiones sobreviven |
| S3 | `SshSession.write` e input loop: capturar `Throwable` (no solo `Exception`) | Escribir en un pipe cerrado nunca tumba la app |

## Historia de usuario

```gherkin
Historia de usuario:
Como usuario de Bastion
Quiero que mis sesiones SSH sigan abiertas cuando cambio a otra app y vuelvo
Para no perder mi trabajo en el terminal cada vez que salgo de la app

Escenario: Cambiar de app y volver (S1/S2)
  Dado que tengo una sesión SSH activa
  Cuando abro otra app y vuelvo a Bastion después de un rato
  Entonces la sesión sigue conectada
  Y puedo seguir escribiendo sin que se cierre ni se caiga la app

Escenario: Conexión inactiva no se corta (S1)
  Dado que tengo una sesión SSH sin actividad
  Cuando pasan varios minutos
  Entonces el heartbeat mantiene viva la conexión

Escenario: Escribir tras un cierre no tumba la app (S3)
  Dado que una sesión murió (red caída)
  Cuando escribo en esa terminal
  Entonces la app registra el error y NO se cierra
```

---

## Criterios de aceptación

| # | Criterio | Verificación |
|---|----------|-------------|
| AC1 | Heartbeat SSH configurado (~30s) + socket keepalive | Sesión inactiva sigue viva tras minutos |
| AC2 | Foreground service activo con notificación mientras haya terminales abiertas | Notificación visible; proceso no muere en background |
| AC3 | El service se detiene cuando se cierran todas las terminales | Sin terminales → sin notificación |
| AC4 | `write` e input loop capturan `Throwable` → nunca crash por pipe cerrado | Matar red y escribir → app viva |

---

## Orden de implementación

1. S1: heartbeat + keepalive en `SshClientManager`.
2. S3: `catch (Throwable)` en `SshSession.write` y el input loop de `TerminalTab`.
3. S2: `SessionKeepAliveService` (foreground) + permisos/manifest + arranque/parada desde `AppLayout`.
4. Build + release `release.sh patch` → v1.1.16.
