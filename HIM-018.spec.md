---

## Contexto

Hoy: `release.sh` crea un release en GitHub **solo con notas** (RHD-BST-003), el APK
real solo vive en el servidor local (`192.168.0.100:8765`), y la app Android chequea
actualizaciones contra ESE servidor local — no contra GitHub. El repo es privado.

El usuario quiere: repo **público**, releases reales en GitHub con el binario adjunto,
un **índice por plataforma** (Android ahora; Desktop/iOS cuando existan) para que la
gente descargue la versión que le sirve, y que el auto-updater de Android valide
contra GitHub si hay una versión mayor.

## Decisiones (confirmadas con el usuario, 2026-07-08)

- **Repo público:** sí, se acepta el riesgo de la API key de Stitch quedando en el
  historial (igual que la password vieja del keystore, ya rotada) — el usuario decidió
  no bloquear por esto.
- **Índice de descargas:** página estática (`docs/index.html`, servida por GitHub
  Pages) que consulta la API de GitHub (`/releases/latest`) **en el cliente** (JS, sin
  build step) y arma los links por plataforma según el prefijo del nombre del asset
  (`bastion-android-*.apk`, `bastion-desktop-*` a futuro). Así no hay que regenerar ni
  commitear nada en cada release — la página siempre refleja el último release real.
- **Auto-update Android:** pasa de consultar `http://192.168.0.100:8765/update` a
  consultar `https://api.github.com/repos/lerna-admin/bastion/releases/latest`.
  Comparación de versión por **semver** (`tag_name` = `vX.Y.Z`) en vez de por
  `versionCode`, ya que GitHub Releases no expone un campo tipo versionCode.

## Cambios

1. **Seguridad primero:** sacar la API key de Stitch hardcodeada de `AGENTS.md`,
   `design/STITCH_DESIGN.md`, `.opencode/stitch-bridge.mjs`, `memoria/datos-operativos.md`
   — referenciar `STITCH_API_KEY` (env var / archivo local), mismo patrón que
   `BASTION_KEYSTORE_PASSWORD`.
2. `release.sh`: `gh release create` adjunta el APK como asset
   (`bastion-android-vX.Y.Z.apk`), no solo notas.
3. `docs/index.html`: página de descargas por plataforma (GitHub Pages).
4. `UpdateChecker.kt` (Android): nueva fuente GitHub, parseo semver, extracción del
   asset Android del array `assets[]`.
5. Repo → público (`gh repo edit --visibility public`).
6. Habilitar GitHub Pages sobre `docs/` en `main`.

## Fuera de alcance
- Firma/verificación de que el APK descargado de GitHub coincide con un hash conocido
  (mitigación adicional de integridad) — se puede agregar después si se quiere más
  robustez; por ahora la firma APK (mismo keystore) sigue siendo la protección real.
- Desktop/iOS en el índice: la página los muestra como "próximamente" hasta que
  existan releases reales para esas plataformas.

## Historia de usuario

```gherkin
Historia de usuario:
Como usuario de Bastion
Quiero descargar la versión de mi plataforma desde una página pública de GitHub
Para no depender de estar en la misma red que el servidor local

Escenario: Descubrir el índice de descargas
  Dado que visito la página de GitHub Pages del proyecto
  Cuando la página carga
  Entonces veo la versión más reciente de Android con su link de descarga directo
  Y veo Desktop/iOS marcados como "próximamente"

Escenario: Auto-update valida contra GitHub
  Dado que tengo Bastion instalado y hay un release más nuevo en GitHub
  Cuando abro la app
  Entonces detecta la versión nueva (comparando semver del tag) y ofrece actualizar
  Y la descarga viene del asset adjunto al release de GitHub
```

## Criterios de aceptación

| # | Criterio | Verificación |
|---|----------|-------------|
| AC1 | Ningún secreto en texto plano en los archivos tocados | grep post-cambio |
| AC2 | `release.sh` sube el APK como asset nombrado por plataforma | `gh release view vX.Y.Z` muestra el asset |
| AC3 | `docs/index.html` público lista Android con link real al último asset | Abrir la URL de Pages |
| AC4 | App detecta y descarga updates desde GitHub (no del server local) | Publicar una versión nueva → la app la detecta |
| AC5 | Repo público, Pages habilitado sobre `docs/` | `gh repo view` + Settings → Pages |
