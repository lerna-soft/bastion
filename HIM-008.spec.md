## Contexto del bug

El usuario reporta que solo puede tener **una** conexión: al abrir una nueva mientras hay una
activa, "reemplaza" la terminal en vez de abrirla en una pestaña nueva. Además falta un botón
explícito para **abrir una pestaña nueva**.

### Análisis del código (cómo se abren y renderizan las pestañas)

Flujo actual (`ui/AppLayout.kt`):
- `terminalSessions = mutableStateListOf<TerminalSession>()` — lista de pestañas.
- `openTerminalSession(host)` → crea un `SshSession` nuevo y hace `terminalSessions.add(...)`.
  **Sí agrega** una pestaña (no reemplaza) y hace `animateScrollToPage(size-1)`.
- `TerminalPagerContent` → `SessionTabBar` (barra de pestañas con título + botón cerrar) +
  `HorizontalPager` con `pageCount = terminalSessions.size`.

Por tanto la lógica de lista/pager es correcta. La causa del "reemplazo" está en el render de
cada página:

- `terminal/TerminalTab.kt:142` → `val webView = remember { webViewCache.getOrPut(session) { createWebView(...) } }`
- `terminal/TerminalTab.kt:148` → `val scope = remember { terminalScopes.getOrPut(session) { ... } }`

Ambos usan `remember { }` **sin `session` como key**. En un `HorizontalPager` los slots de página
se reciclan; cuando un slot se reutiliza para otra `session` (al cerrar una pestaña e desplazar
índices, o al reciclar), el `remember` retiene el `WebView`/`scope` de la sesión anterior porque
no se reevalúa al cambiar `session`. Resultado: la página muestra la terminal de otra sesión →
se percibe como "reemplazo".

Falta además un botón "+" (nueva pestaña) en `SessionTabBar`.

## Hallazgos

| # | Severidad | Hallazgo | Archivo:línea |
|---|-----------|----------|---------------|
| FA | **Alta** | `remember { webViewCache.getOrPut(session){} }` sin key `session`: al reciclar slots del pager la página conserva el WebView de otra sesión → "reemplazo" visual. | `TerminalTab.kt:142` |
| FB | **Alta** | Igual problema con `remember { terminalScopes.getOrPut(session){} }` (scope de I/O de otra sesión). | `TerminalTab.kt:148` |
| FC | Media | No existe botón "Nueva pestaña" en la barra; la única forma de abrir es desde el vault ("Connect" en un host). | `AppLayout.kt` SessionTabBar |
| FD | Baja | `webViewCache`/`terminalScopes` no se limpian del todo al cerrar (verificar fuga de WebView tras cerrar y reabrir). | `AppLayout.closeTerminalSession` |

## Historia de usuario

```gherkin
Historia de usuario:
Como usuario de Bastion
Quiero abrir varias conexiones SSH en pestañas independientes
Para trabajar en múltiples servidores a la vez sin perder ninguna sesión

Escenario: Abrir una segunda conexión no reemplaza la primera (FA/FB)
  Dado que tengo una sesión activa al servidor A en la pestaña 1
  Cuando abro una conexión al servidor B
  Entonces se crea la pestaña 2 con el servidor B
  Y la pestaña 1 (servidor A) sigue activa y muestra su propia terminal
  Y al volver a la pestaña 1 veo la terminal del servidor A, no la del B

Escenario: Botón "Nueva pestaña" (FC)
  Dado que estoy en la vista de sesiones con al menos una pestaña
  Cuando pulso el botón "+" de la barra de pestañas
  Entonces se me ofrece elegir un host del vault para abrir en pestaña nueva
  Y al elegirlo se abre en una pestaña adicional (no reemplaza la actual)

Escenario: Cerrar una pestaña conserva las demás (FA/FD)
  Dado que tengo 3 pestañas abiertas (A, B, C)
  Cuando cierro la pestaña B
  Entonces quedan las pestañas A y C con sus terminales correctas
  Y ninguna muestra el contenido de la pestaña cerrada
```

## Criterios de aceptación

| # | Criterio | Verificación | Hallazgo |
|---|----------|-------------|----------|
| AC1 | `webView` y `scope` se recuerdan con `remember(session) { ... }` (key = session) | Abrir 2 sesiones y alternar: cada pestaña muestra su terminal | FA, FB |
| AC2 | Abrir una conexión con otra activa crea pestaña nueva (ya funciona) y NO reemplaza | 2+ pestañas visibles en SessionTabBar | FA |
| AC3 | Botón "+" en SessionTabBar abre un selector de host y agrega pestaña | Pulsar "+" → elegir host → nueva pestaña | FC |
| AC4 | Al cerrar una pestaña, las restantes conservan su terminal y no hay fuga de WebView | Cerrar la del medio de 3 → las otras intactas | FA, FD |

## Plan de tests (test-master)

El fix central (FA/FB) es comportamiento Compose/WebView (recomposición + reciclado de slots del
`HorizontalPager`) y se valida **en dispositivo** vía AC1-AC4 — no es aislable en un unit test sin
Robolectric/Compose-UI-test. La lógica de la lista de pestañas vive en funciones locales del
composable y usa `TerminalSession` (que es `private` y contiene un `SshSession` con dependencias
Android), por lo que testearla exigiría un refactor invasivo que no aporta valor real (sería
reimplementar la lista, un anti-patrón de test-master).

**Decisión:** el gate de RHD-BST-011 se cumple con la suite existente (HostValidator, AppSettings,
UpdateChecker — todos verdes) más `assembleDebug` OK. La validación funcional de multi-pestaña es
manual en dispositivo (AC1-AC4). Si en el futuro se extrae la gestión de pestañas a un modelo puro
(`TabSessionManager`), se añadirán T1-T3 sobre ese modelo.
