## Contexto del bug

Tras liberar v1.1.9, la app (instalada en v1.1.8, code 20) muestra el aviso "Update available"
pero al pulsar **"Update now"** la instalación falla y **no llega ningún log nuevo al servidor**
(`~/apk-share/bastion-logs.ndjson` se detiene a las 03:47, aún en code 20). Sin logs del intento
fallido, el diagnóstico queda a ciegas — de ahí que este spec incluya primero **cómo analizar el
flujo** y **cómo recuperar visibilidad de logs** antes de arreglar.

Descartado por evidencia (validación test-master, 2026-07-08):
- **Firma NO es la causa**: `apksigner` confirma que v1.1.6, v1.1.8 y v1.1.9 comparten
  el mismo certificado (SHA-256 `8172620498e7…1ba2`, `CN=Bastion`) y el mismo esquema (solo v2).
  No hay signature mismatch ni cambio de keystore.

---

## Cómo analizar el código (mapa del flujo — leer en este orden)

El flujo de auto-update recorre 4 archivos. Para entenderlo, seguirlos en secuencia:

1. `BastionApp.kt`
   - `onCreate()` → `checkForUpdate()` al arrancar.
   - `checkForUpdate()` → llama `UpdateChecker.checkForUpdate(serverUrl)`, emite `UpdateState.Available` / `Idle` / `Error`.
   - `downloadUpdate(info)` → `UpdateChecker.downloadApk(...)`; al terminar emite `Ready` y llama `installApk(...)`.
2. `update/UpdateChecker.kt`
   - `checkForUpdate()` → GET `/update`, compara `versionCode` server vs `BuildConfig.VERSION_CODE`.
   - `downloadApk()` → descarga a `cacheDir/updates/<fileName>`, callback de progreso.
   - `installApk()` → `FileProvider.getUriForFile(...)` + intent `ACTION_VIEW` `application/vnd.android.package-archive`.
3. `MainActivity.kt` (líneas 86-183)
   - Observa `app.updateState`; dibuja diálogos `Available` / `Downloading` / `Ready`.
   - Botón "Update now" → `app.downloadUpdate(state.info)`.
   - Botón "Install" (diálogo Ready) → toma un archivo del dir `updates` y llama `installApk`.
4. `logging/RemoteLogger.kt`
   - `e()`/`crash()` → `enqueue()` (emite a `SharedFlow` async) + `forceFlush()`.
   - `AndroidManifest.xml`: permiso `REQUEST_INSTALL_PACKAGES` declarado; `FileProvider` con `cache-path updates/`.

Comando de verificación de firma (para reproducir el descarte):
`JAVA_HOME=~/dev-tools/jdk-17.0.19+10 ~/android-build-env/android-sdk/build-tools/36.0.0/apksigner verify --print-certs <apk>`

## Hallazgos de la validación (test-master)

| # | Severidad | Hallazgo | Archivo:línea |
|---|-----------|----------|---------------|
| F1 | **Alta** | `installApk()` lanza el intent de instalación sin verificar `packageManager.canRequestPackageInstalls()`. Si el permiso "Instalar apps desconocidas" para Bastion está revocado (Android puede resetearlo), el sistema bloquea la instalación y regresa a la app **sin lanzar excepción** → "Update now" parece no hacer nada y no se loguea error. | `UpdateChecker.kt:114-132` |
| F2 | **Alta** | Condición de carrera en `RemoteLogger.e()`/`crash()`: `enqueue()` emite al `SharedFlow` de forma asíncrona y `forceFlush()` lee `buffer` que puede aún no contener la entrada → los errores de descarga/instalación **no llegan al servidor** justo cuando fallan. Explica el "no llegan logs". | `RemoteLogger.kt:146-166` |
| F3 | **Media** | `downloadApk()` no valida integridad: no compara `outFile.length()` contra `updateInfo.fileSize` ni checksum. Si el server se reinicia a mitad de descarga (ocurrió a las 08:25 al construir v1.1.9), el APK queda truncado → "parse error / paquete corrupto" al instalar. | `UpdateChecker.kt:68-112` |
| F4 | **Media** | Botón "Install" del diálogo Ready usa `File(cacheDir,"updates").listFiles()?.firstOrNull()`. El dir **nunca se limpia** (se acumulan 8+ APKs de v1.1.1…v1.1.8); `firstOrNull()` devuelve un archivo arbitrario, posiblemente una versión vieja → instala APK equivocado/antiguo. | `MainActivity.kt:167-170` |
| F5 | Baja | APKs v1.1.7+ quedaron `root:root` (build en Docker como root). No rompe el serve (world-readable) pero es un smell del pipeline de build. | `~/apk-share/` |

## Historia de usuario

```gherkin
Historia de usuario:
Como usuario de Bastion
Quiero que al pulsar "Update now" la nueva versión se instale de forma confiable
Para mantener la app actualizada sin quedarme atascado en una versión vieja

Escenario: Permiso de instalación no concedido (F1)
  Dado que el permiso "Instalar apps desconocidas" para Bastion está revocado
  Cuando pulso "Update now" y la descarga termina
  Entonces la app detecta que no puede instalar
  Y me redirige a la pantalla de ajustes para conceder el permiso (ACTION_MANAGE_UNKNOWN_APP_SOURCES)
  Y registra un log "install blocked: canRequestPackageInstalls=false"

Escenario: Error de instalación siempre queda registrado (F2)
  Dado que la instalación o descarga falla por cualquier motivo
  Cuando ocurre el error
  Entonces el log de ERROR se envía al servidor de forma síncrona (o se persiste y reintenta)
  Y aparece en ~/apk-share/bastion-logs.ndjson

Escenario: Descarga corrupta detectada (F3)
  Dado que la descarga se interrumpe y el archivo queda incompleto
  Cuando termina el intento de descarga
  Entonces la app compara el tamaño descargado contra fileSize esperado
  Y si no coincide, borra el archivo, muestra "download corrupt" y no lanza el instalador

Escenario: Instalar siempre el APK recién descargado (F4)
  Dado que existen varios APKs viejos en cacheDir/updates
  Cuando se instala la actualización
  Entonces se instala exactamente el archivo recién descargado (por nombre de fileName)
  Y los APKs anteriores se eliminan del cache
```

## Criterios de aceptación

| # | Criterio | Verificación | Hallazgo |
|---|----------|-------------|----------|
| AC1 | Antes de instalar, se verifica `canRequestPackageInstalls()`; si es false, se abre `ACTION_MANAGE_UNKNOWN_APP_SOURCES` para Bastion | Revocar permiso en dispositivo → "Update now" → aparece pantalla de permiso | F1 |
| AC2 | `RemoteLogger.e()`/`crash()` garantizan entrega del log (flush síncrono del propio entry o persistencia+reintento inmediato) | Forzar un error → aparece en bastion-logs.ndjson en <5s | F2 |
| AC3 | `downloadApk()` valida `outFile.length() == updateInfo.fileSize` (>0); si no coincide, borra y reporta error | Cortar descarga a mitad → estado Error, sin lanzar instalador | F3 |
| AC4 | La instalación usa el `File(updatesDir, updateInfo.fileName)` exacto; el dir `updates` se limpia antes de cada descarga | Con APKs viejos en cache → instala solo el nuevo | F4 |
| AC5 | Logs nuevos del intento de update llegan al servidor y permiten confirmar la causa raíz real en dispositivo | Reinstalar build con fix F2 → reproducir → revisar logs | F1,F2 |

## Orden de implementación sugerido

1. **F2 primero** (recuperar visibilidad de logs) — sin esto seguimos ciegos ante el fallo real en dispositivo.
2. **F1** (gate de permiso de instalación) — causa más probable del "no pasa nada".
3. **F3** y **F4** (robustez de descarga/instalación).
4. Tests T1-T5 + build release vía `release.sh patch` (RHD-BST-002) → v1.1.10.

> Nota: F1 es la hipótesis más probable, pero **AC5 (recuperar logs, F2) debe ir primero** para
> confirmar la causa raíz real en el dispositivo antes de dar por cerrado el bug.
