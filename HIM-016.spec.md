## Visión

Bastion pasa de ser una app solo-Android a un producto multiplataforma: Android (ya
estable, no se toca), Windows, Linux, Mac (vía Compose Desktop, un mismo target) y,
a futuro, iOS (bloqueado técnicamente, ver ADR-D4).

Esta iteración es un **walking skeleton**: la porción más delgada posible que prueba,
de punta a punta, que la arquitectura elegida funciona — sin intentar portar todavía
el vault, Settings, multi-pestaña, etc. Eso queda para HIM-017+.

## Decisiones de arquitectura (discutidas y confirmadas con el usuario, 2026-07-08)

### ADR-D1 — Estructura de módulos: `:core` como Kotlin/JVM plano, no KMP formal (todavía)

**Contexto:** hay que compartir la lógica SSH (Apache MINA SSHD + BouncyCastle) entre
Android y Desktop.

**Decisión:** `:core` se crea como módulo **Kotlin/JVM plano** (`kotlin("jvm")`), NO
como módulo Kotlin Multiplatform formal (sin `expect`/`actual`, sin
`kotlin-multiplatform` plugin todavía). Tanto Android (`:app`) como Desktop
(`:desktopApp`) son JVM real — Apache MINA SSHD corre idéntico en ambos sin
necesidad de abstracción de plataforma.

**Por qué:** simplifica muchísimo esta fase. La formalización a KMP con
`expect`/`actual` (necesaria para que iOS use una librería SSH distinta, ver ADR-D4)
se hace **cuando llegue esa fase**, no antes — evita complejidad prematura.

**Alternativa descartada:** módulo KMP formal desde ya (`androidTarget()` +
`jvm("desktop")`) — más ceremonia sin beneficio inmediato, ya que ambos targets son
JVM idéntico en esta fase.

### ADR-D2 — Terminal en desktop: reusar xterm.js vía navegador embebido (JCEF)

**Decisión:** el terminal de `:desktopApp` reutiliza **tal cual** los assets
`app/src/main/assets/terminal/{index.html,xterm.js,xterm-addon-fit.js,xterm.css}`
ya probados en Android, cargados en un WebView de escritorio basado en JCEF
(Chromium embebido — librería `dev.datlag.kcef` o `compose-webview-multiplatform`).

**Trade-off aceptado:** JCEF agrega ~150-200MB al instalador (Chromium embebido).
Se acepta a cambio de reusar el 100% del trabajo de temas/fit/comandos rápidos ya
hecho y probado, en vez de reescribir el terminal con una librería nativa
(JediTerm) que perdería ese trabajo.

### ADR-D3 — Secretos en desktop: keychain nativo del SO

**Decisión:** los secretos (password/private key) se guardan en el keychain nativo
de cada SO — Linux: Secret Service (`libsecret`, vía `org.freedesktop.secret` D-Bus
API); Windows: Credential Manager (DPAPI) — fase futura; macOS: Keychain — fase
futura. Mismo nivel de seguridad que Android Keystore (HIM-001 ADR-004).

**Alcance de esta iteración:** el vault persistente (guardar hosts + keychain) queda
**fuera de alcance** — ver "Fuera de alcance" abajo. Esta ADR queda documentada para
cuando se implemente en HIM-017.

### ADR-D4 — iOS bloqueado técnicamente (informativo, no bloquea esta iteración)

Apache MINA SSHD es JVM puro — **no corre en Kotlin/Native**, el target que exige
iOS. El día que se aborde iOS, la capa SSH necesita una implementación
completamente distinta (librería Swift nativa vía interop, p.ej. algo tipo
Citadel/NIOSSH), detrás de un `expect`/`actual` en `:core` ya formalizado como KMP.
No es una tarea de "portar", es un componente nuevo. Se documenta para que la
decisión de invertir en iOS se tome con esta información.

### ADR-D5 — Secuenciación: Linux primero, end-to-end

**Decisión:** esta iteración prueba TODO en Linux (este servidor): compila, corre,
conecta SSH real, renderiza terminal, genera instalable (`.deb`/AppImage). Windows
queda para una spec aparte (HIM-018) que configura GitHub Actions con runner
`windows-latest`, porque generar un `.exe/.msi` normalmente requiere compilar en esa
plataforma (o CI). Mac queda para más adelante (mismo target de código que
Windows/Linux, pero notarización requiere cuenta Apple Developer + macOS real).

## Fuera de alcance de esta iteración (deferred a HIM-017+)

- Vault persistente (guardar hosts, Room/SQLite en desktop)
- Keychain real (ADR-D3 documentada pero NO implementada aún — el password se
  mantiene solo en memoria de la sesión para esta iteración)
- Multi-pestaña (HIM-008/HIM-013 portados)
- Settings, SSH Keys UI, Logs UI, auto-update
- Windows y Mac (ADR-D5)

## Historia de usuario

```gherkin
Historia de usuario:
Como desarrollador de Bastion
Quiero un walking skeleton de Bastion Desktop que corra en Linux
Para validar que la arquitectura elegida (core JVM compartido + JCEF + Compose Desktop)
funciona de punta a punta antes de invertir en portar el resto de features

Escenario: Extraer :core sin romper Android
  Dado que SshSession/SshClientManager/AuthMethods viven hoy en :app
  Cuando se mueven a :core (módulo Kotlin/JVM plano)
  Entonces :app sigue compilando y funcionando en Android exactamente igual (regresión cero)

Escenario: Levantar el skeleton de :desktopApp
  Dado el nuevo módulo :desktopApp (Compose Multiplatform Desktop)
  Cuando se ejecuta en este servidor Linux
  Entonces se abre una ventana con una pantalla mínima de "Conectar"
  (host, puerto, usuario, password — sin persistencia)

Escenario: Conexión SSH real desde desktop
  Dado que ingreso host/usuario/password de un servidor real
  Cuando presiono "Conectar"
  Entonces se abre una sesión SSH real usando :core (mismo SshSession que Android)

Escenario: Terminal funcional en desktop
  Dado que la sesión SSH está activa
  Cuando interactúo con el terminal embebido (JCEF + xterm.js)
  Entonces veo la salida del shell remoto y puedo escribir comandos
  Y el comportamiento es equivalente al terminal de Android (mismos assets)

Escenario: Instalable de Linux generado
  Dado que el walking skeleton funciona
  Cuando corro la tarea de empaquetado de Compose Desktop
  Entonces se genera un .deb (o AppImage) instalable
  Y al instalarlo y abrirlo, la app funciona igual que en modo desarrollo
```

## Criterios de aceptación

| # | Criterio | Verificación |
|---|----------|-------------|
| AC1 | `:core` módulo Kotlin/JVM creado; `SshSession`/`SshClientManager`/`AuthMethods` movidos ahí | `./gradlew :core:compileKotlin` |
| AC2 | `:app` (Android) sigue compilando y pasando su suite de tests sin cambios de comportamiento | `./gradlew :app:testReleaseUnitTest :app:assembleRelease` (regresión) |

## Plan de implementación

1. Reestructurar `settings.gradle.kts`: agregar `:core` y `:desktopApp`.
2. Crear `:core` (Kotlin/JVM), mover `ssh/SshSession.kt`, `ssh/SshClientManager.kt`,
   `ssh/AuthMethods.kt` (y sus dependencias directas) desde `:app`. Actualizar `:app`
   para depender de `:core` y ajustar imports (sin cambiar comportamiento).
3. Validar AC1+AC2 (regresión Android) antes de tocar nada de desktop.
4. Crear `:desktopApp` con plugin `org.jetbrains.compose` + `org.jetbrains.kotlin.jvm`.
   Pantalla mínima de conexión + integración JCEF para el terminal (reusando los
   assets de `app/src/main/assets/terminal/` copiados a `desktopApp/src/main/resources/`).
5. Validar AC3-AC6 en este servidor.
6. Configurar tarea de empaquetado Compose Desktop (`nativeDistributions`) para `.deb`.
7. Validar AC7.

## Riesgos

- **JCEF en Linux headless/servidor:** este servidor puede no tener entorno gráfico
  (X11/Wayland) para correr una ventana Compose Desktop de verdad. Se valida temprano
  (paso 3-4) — si no hay display, la verificación visual de AC3-AC6 requiere ejecutar
  en una máquina Linux con GUI, o usar Xvfb (framebuffer virtual) para al menos
  confirmar que la app arranca sin crashear.
- **Tamaño de JCEF:** ~150-200MB agregados al instalable — aceptado en ADR-D2, pero
  vale confirmarlo una vez armado.
